# Zenn 記事レビュースキル

指定された記事ファイルをレビューし、文体・構成・技術的正確性についてフィードバックを返します。

## 手順

### 1. 対象記事を読む

`$ARGUMENTS` で指定されたファイルを読みます。指定がない場合は、`articles/` 配下で最も新しい未公開記事（`published: false`）を対象とします。

### 2. 過去記事から文体を把握する

`articles/` 配下の公開済み記事（`published: true`）から直近3件を読み、文体パターン（見出し名、文末表現、導入・締めの書き方など）を把握します。

### 3. 以下の観点でレビューする

#### タイトルと見出し、内容の整合性

- **内容との一致**: タイトルと見出しが記事の内容を適切に表しているか
- **誇張がないか**: タイトルや見出しが過度に誇張されていないか

#### 文体チェック

過去記事と照らし合わせ、以下の点を確認します。

- **見出し名の統一**: 過去記事で使われている見出し名（例: 「はじめに」「おわりに」「参考」）と一致しているか
- **文末表現**: 「です/ます」調で統一されているか
- **導入文**: 過去記事の導入パターン（背景・動機の説明）に沿っているか
- **締めの文**: 過去記事の締めパターン（例: 「本記事の内容が、参考になれば幸いです。」）に沿っているか
- **箇条書きと文章のバランス**: 箇条書きだけで構成されたセクションがないか。過去記事では文章主体で箇条書きは補助的に使われている
- **指示語の明確さ**: 「この」「あの」「その」などの曖昧な指示語を避け、「上記の」「前述の」「下記の」「以下の」など、参照先が明確な表現を使っているか

#### 構成チェック

- **セクションの粒度**: 見出しが細かすぎないか（内容が1〜2行しかないセクションは統合を検討）
- **流れの自然さ**: セクション間の接続が自然か
- **概念の回収**: 記事の導入で使った概念が本文で回収されているか
- **冗長な説明がないか**: 同じことを繰り返し説明していたり、一文で済む内容を複数文で回りくどく書いていないか。読者が理解するのに必要十分な分量になっているか
- **焦点がぶれていないか**: 記事のテーマや各セクションの主題と直接関係ない話題が混ざっていないか。脱線や余談が読者の理解を妨げていないか

#### 独自の価値チェック

公式ドキュメントの内容をそのまま繰り返すだけでは、読者が記事を読む理由がない。記事が公式ドキュメントに対して+αの価値を提供できているかを確認する。

- **読みやすさの向上**: 公式ドキュメントの説明を噛み砕いたり、具体例を交えることで、より理解しやすくなっているか
- **情報の整理・集約**: 公式ドキュメントの複数ページに散らばっている関連情報が、1つの記事にまとまっているか
- **独自の視点・解釈**: 筆者自身の言葉で概念を再構成したり、公式にはない切り口（比較表、段階的な説明、実務的な観点など）で整理されているか
- **公式の焼き直しになっていないか**: セクション単位で見て、公式ドキュメントの引用・要約だけで構成されている箇所がないか。引用は根拠として使い、本文は筆者自身の説明で書く

#### 技術的正確性チェック

- **APIやライブラリのバージョン情報**: 記事で言及しているバージョンが正しいか、Web検索で最新情報を確認する
- **コードブロック**: 言語指定（例: ` ```tsx:説明 `）が付いているか
- **公式ドキュメントの引用**: 引用がある場合、参照リンクが正しいか
- **引用の参照URL**: 記事内で `>` を使った引用（blockquote）がある場合、その直後に `> 参照：URL` の形式で参照元URLが記載されているか確認する。URLが欠けている引用があれば指摘する

#### フォーマットチェック

- **参考文献セクション**: 末尾に「参考」セクションがあり、関連リンクがまとめられているか
- **リンクの一貫性**: 記事全体でリンクのドメインが統一されているか（例: `ja.react.dev` と `react.dev` が混在していないか）
- **frontmatter**: `topics` が適切か

### 4. レビュー結果を出力する

以下のフォーマットでレビュー結果を出力します。問題がない項目は省略して構いません。

```
## レビュー結果: [記事タイトル]

### タイトルと見出し
- [指摘事項と修正案]

### 文体
- [指摘事項と修正案]

### 構成
- [指摘事項と修正案]

### 独自の価値
- [指摘事項と修正案]

### 技術的正確性
- [指摘事項と修正案]

### フォーマット
- [指摘事項と修正案]

### 修正・加筆の提案
- [具体的な修正箇所と修正案、または加筆すべき箇所とその理由]

### 総評
[全体的な所感を1〜2文で]
```
